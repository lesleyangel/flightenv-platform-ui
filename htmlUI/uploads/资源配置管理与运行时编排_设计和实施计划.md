# FlightEnv 资源配置管理与运行时编排设计和实施计划

日期：2026-06-05

> 2026-06-08 状态更新：本文保留为资源配置管理、资产锁定、run evidence 和执行层验收的阶段总方案。动态编排主入口已升级为 [状态方程与QoI动态编排_统一设计.md](./状态方程与QoI动态编排_统一设计.md)。本文中固定 `Sensor -> Filter -> Trajectory -> Field -> Damage -> Life` 多节点链路不再作为后续模型编排默认主架构，应按“算子图的 ROS2/exe 执行适配或兼容验证链路”解释。

## 1. 执行结论

本方案用于收口 `FlightEnvMultiRepo` 后续平台化开发中的两个核心问题：

```text
资源配置管理：用什么资源、模型、数据、依赖和版本。
运行时编排：这些资源在在线/离线/回放模式下怎么跑。
```

两者必须分开设计、协同实现。资源配置管理不直接承担高频调度；运行时编排不拥有模型资产语义。前者负责登记、校验、锁定和追溯，后者负责启动、订阅、定时、缓存、发布、落盘和诊断。

最终目标不是做大而全的通用数字孪生平台，而是形成：

> 面向复杂工程对象的模型驱动数字孪生运行平台。

平台最终应交付一个可运行、可回放、可审计的工程对象数字孪生画像。给定一个对象或一次 run，系统能够解释：

```text
对象是谁
传感器绑在哪
当前状态是什么
未来弹道和物理场如何演化
用了哪些模型、版本和配置
寿命/损伤/首超破坏时间结论来自哪些证据
结果能否回放和复现
失败或无法评估的原因是什么
```

推荐主线是：

```text
先稳定弹道核心和模型包
  -> 再做 trajectory-first 资源元数据和运行证据
  -> 再做运行时编排 MVP
  -> 再打通弹道/多场/损伤/寿命开环链路
  -> 再做 workflow validator 和模型资产平台化
  -> 最后再进入耦合弹道规划、实验管理和高级编排
```

第一阶段不要做通用 DAG executor、全量平台 UI、旧任务库推倒重写、在线寿命反哺弹道闭环规划。

## 2. 多 Agent 评估收敛结论

本方案基于三个独立方向的评估收敛：

| 评估方向 | 核心结论 | 写入本方案的位置 |
| --- | --- | --- |
| 最终效果和平台目标 | 应把“数字孪生画像”作为一级交付物；平台价值是可信、可复现、可解释、可审计 | 第 3 节、第 13 节 |
| 模块拆分和边界解耦 | 资源/编排实现主体在 `flightenv-runtime-private`；公共 DTO/API 进 `flightenv-contracts`；UI 只经 SDK 消费 | 第 6 节、第 7 节 |
| 实施和验收 | 先稳弹道 artifact，再做轻量元数据、run 证据、编排节点，多场/寿命后置 | 第 10 节、第 11 节 |

当前已有文档结论与本方案一致：

- `doc/design/工程对象数字孪生运行平台_分期规划设计报告.md` 已明确平台定位和大阶段。
- `doc/design/数据管理平台_设计方案.md` 已定义 `ModelDefinition`、`ModelPort`、`ModelRelation`、`RunCatalog`、`HealthLedger`。
- `doc/design/弹道模块_运行时节点与工具链设计.md` 已明确弹道采用 Core lib、ROS2 node、CLI 三层形态。

本方案不是替代这些文档，而是把它们收口成一条资源配置管理和运行时编排的执行路线。

## 3. 最终效果

### 3.1 产品最终形态

最终系统应支持三类运行方式：

```powershell
flightenv validate --workflow workflows/reentry_life_chain.json
flightenv run --case cases/reentry_online.json --profile online
flightenv replay --run _local_artifacts/online-runs/run_20260605_153000
```

其中：

- `validate` 做资源、模型、端口、单位、topic、依赖和输出目录的启动前校验。
- `run` 启动在线链路，各模块按自己的频率运行。
- `replay` 不启动 ROS2 节点，直接打开历史 run package 做回放、诊断和报告。

UI 最终应从实时看板升级为 Twin Workbench，至少包含：

| 页面/视图 | 能力 |
| --- | --- |
| 在线运行 | 当前状态、滤波状态、弹道预测、多场摘要、寿命/风险实时显示 |
| 对象画像 | 对象树、部件、区域、测点、传感器、材料和模型绑定 |
| 模型资产 | 模型 ID、版本、checksum、适用包络、端口、校验报告 |
| 历史回放 | 打开 run、读取帧摘要、查看状态/轨迹/场/寿命结果 |
| 健康账本 | 对象损伤累计、RUL、首超破坏时间、证据来源和维护修正 |
| 诊断报告 | 配置错误、模型包错误、单位/端口不匹配、节点超时和降级原因 |

### 3.2 在线开环主链路

近期目标链路应保持开环：

```text
SensorNode
  -> FilterNode
  -> StateEstimateFrame
  -> TrajectoryNode
  -> TrajectoryPredictionFrame
  -> FieldPredictionNode
  -> FieldPredictionFrame
  -> DamageNode
  -> DamageAssessmentFrame
  -> LifeNode
  -> LifeAssessmentFrame
  -> UI / Archive / HealthLedger
```

这条链路的原则是：

- 每个模块有自己的频率。
- 上游不阻塞下游。
- 下游消费带时间戳和来源的最新有效数据。
- 所有输出可落盘和回放。
- 模型之间只通过 DTO 交互，不互相 include 私有实现。

建议初始频率：

| 模块 | 建议频率 | 说明 |
| --- | --- | --- |
| 传感器/滤波 | 20-50 Hz | 保持当前状态估计实时性 |
| 弹道预测 | 5-20 Hz | 根据最新状态输出一条未来轨迹 |
| 多场预测 | 1-5 Hz | 先做摘要或代理模型，避免阻塞主链 |
| 损伤/寿命 | 0.5-2 Hz | 输出风险、RUL、首超破坏时间和无法评估原因 |
| UI | 10-30 Hz 显示刷新 | 不作为模型运行驱动 |
| Archive | 异步 | 不阻塞实时链路 |

### 3.3 数字孪生画像验收

最终每个 run 至少应生成一份可读画像摘要：

```text
TwinProfile
  object_id
  mission_id
  run_id
  state_source
  trajectory_model_snapshot
  field_model_snapshot
  damage_model_snapshot
  health_model_snapshot
  input_data_refs
  output_archive_refs
  health_summary
  failure_or_unknown_reasons
```

验收标准：

```text
给定一个历史 run，用户可以打开对象画像，并解释：
1. 这次运行对应哪个工程对象和任务。
2. 使用了哪些传感器、模型、配置和版本。
3. 当前状态、未来弹道、多场预测和寿命结论是什么。
4. 结论对应哪些输入帧、模型版本和证据文件。
5. 无法评估或失败时的明确原因。
```

## 4. 总体设计原则

### 4.1 资源配置管理和运行时编排分离

```text
Resource Configuration Management
  负责：模型、数据、资源、依赖、配置、绑定、版本、checksum、适用范围、run 证据。

Runtime Orchestration
  负责：启动、订阅、发布、定时、缓存、超时、降级、落盘、健康监控、关闭。
```

不要让数据库层执行模型推理，也不要让运行时节点私自决定模型资产语义。

### 4.2 元数据入库，大数据走 artifact

平台库只存元数据、引用、摘要和索引：

```text
platform.sqlite
  TwinObject
  SensorChannel
  ModelDefinition
  ModelPort
  ModelBinding
  ModelRelation
  DataAssetRef
  RunCatalog
  HealthLedger
```

大数据和生成物放 artifact：

```text
_local_artifacts/
  online-runs/<run_id>/
  model-packages/
  flightenv-trajectory/trajectory-package/
  reports/
```

旧任务/场数据库继续作为数值任务数据仓，不把 `TwinObject`、`ModelDefinition`、`RunCatalog`、`HealthLedger` 塞回旧 schema。

### 4.3 Contracts-first

影响多仓的中间对象必须先进入 `flightenv-contracts`：

```text
StateEstimateFrame
TrajectoryPredictionFrame
FieldPredictionFrame
DamageAssessmentFrame
LifeAssessmentFrame
ModelAssetDTO
ModelPortDTO
WorkflowDefinitionDTO
RunManifestDTO
ResourceLockDTO
HealthLedgerDTO
DiagnosticFrame
```

DTO 必须只表达跨模块语义，不暴露 Core、DB、ROS2 node 或 legacy solver 内部类型。

### 4.4 每期必须有可演示闭环

每期都要有一个真实 case 验收，不允许只增加 schema、文档或抽象层：

```text
弹道核心稳定期：当前状态 -> 未来轨迹 -> CLI/Tests/smoke。
运行证据期：run -> manifest -> model snapshot -> replay。
编排期：filter state -> trajectory node -> trajectory topic/archive。
开环链路期：trajectory -> field mock -> damage mock -> life mock -> UI/archive。
```

## 5. 资源配置管理设计

### 5.1 管理对象

资源配置管理要覆盖一次运行所需的全部输入和约束：

| 资源类别 | 示例 | 管理重点 |
| --- | --- | --- |
| 模型包 | 弹道、多场、损伤、寿命、mapper | model_id、version、checksum、适用包络、端口 |
| 数据资产 | 任务库、run package、材料参数、结构对象 | uri、schema_version、checksum、引用关系 |
| 运行环境 | DLL/lib/exe、ROS2 node、Python/ONNX、第三方依赖 | runtime_type、artifact_ref、依赖清单 |
| 通信资源 | topic、QoS、DTO、频率、超时 | topic_map、max_input_age、deadline |
| 存储资源 | run 目录、frame 格式、索引库、日志 | storage_policy、archive policy |
| 计算资源 | CPU/GPU、线程、进程、在线/离线模式 | runtime_profile、resource_limit |
| 诊断资源 | validator、baseline、report、golden case | diagnostic_policy、validation_ref |

### 5.2 核心对象

#### ResourceRef

统一资源引用，不复制资源本体：

```text

schema_version

description
```

描述模型资产：

```text

runtime_type = lib | dll | exe | ros2_node | python | onnx
version
artifact_ref
checksum
applicable_envelope_json
validation_ref
status
```

#### ModelPort

描述模型输入输出：

```text
port_id
model_id
direction = input | output
variable_name
data_type
quantity
unit
required
schema_id
```

典型端口：

```text

TrajectoryModel.output.future_trajectory  -> TrajectoryPredictionFrame

AeroThermalModel.output.field_prediction  -> FieldPredictionFrame
```

#### ModelRelation

描述模型关系，不描述私有调用：

```text

source_port
target_model_id

interface_schema
mapper_model_id
required
max_lag_ms
description
```

#### WorkflowDefinition

描述本次运行链路：

```text

profile = online | replay | batch | validation | coupled_planning

edges[]
storage_policy
diagnostic_policy
fallback_policy
```

每个 workflow node：

```text

node_type = ros2_node | inproc_component | cli | batch_tool | external_service
rate_hz
input_ports
output_ports
max_input_age_ms

fallback_policy

```

```text

target_node.input_port
dto_type

required

```

#### RuntimeProfile

描述运行模式和资源限制：

```text
profile_id

realtime_required
default_qos
threading_policy
process_policy
archive_enabled
resource_limits
```

`RunManifest` 是本次运行实际发生了什么；`ResourceLock` 是本次运行锁定了哪些资源版本。


## 6. 运行时编排设计

### 6.1 编排职责

运行时编排器负责：

```text
启动前：
  读取 WorkflowDefinition
  解析 ResourceRef
  校验模型包、DTO、单位、topic、频率、依赖
  生成 RunManifest 和 ResourceLock

运行中：
  启动节点或进程
  管理 topic 映射
  缓存最新状态和结果
  按各模块频率触发计算
  监控延迟、丢帧、超时、失败和降级
  异步落盘和输出诊断

运行后：
  flush archive
  关闭节点
  固化 run package
  写 RunCatalog
  生成 summary 和 report
```

### 6.2 编排器不做什么

编排器不负责：

- 训练模型。
- 实现弹道、多场、损伤或寿命算法。
- 直接访问 UI 控件。
- 直接解析 private legacy payload 给下游。
- 替代 `flightenv-trajectory` 的核心预测 API。
- 在第一阶段做通用 DAG executor。

### 6.3 在线数据缓存和新鲜度

各模块之间不做强同步阻塞。下游读最新有效帧：

```text
LatestFrameCache<T>
  latest_frame
  stamp_ns
  source_node_id
  sequence
  validity
  diagnostic
```

每条输入边必须定义：

```text
max_input_age_ms
required
fallback_policy
```

例如：

| 输入 | max age | 超时策略 |
| --- | --- | --- |
| 弹道消费状态估计 | 200 ms | 输出 stale diagnostic，继续使用上次或暂停预测 |
| 多场消费弹道 | 1000 ms | 使用最近一条轨迹或输出 degraded |
| 寿命消费多场 | 3000 ms | 输出 unknown/reason，不写伪 RUL |

### 6.4 运行时节点形态

近期推荐：

```text
EnvNodeFilter
FlightEnvTrajectoryNode
FlightEnvFieldPredictionNode
FlightEnvDamageNode
FlightEnvLifeNode
EnvNodeController
ArchiveWriter
```

其中 `FlightEnvTrajectoryNode` 只负责 ROS2/DTO 转换、缓存、定时器、发布和诊断；算法仍通过：

```cpp
flightenv::trajectory::CreateTrajectoryPredictor()->Predict(request)
```

### 6.5 开环和耦合的边界

近期在线主链必须保持开环：

```text
状态 -> 弹道 -> 多场 -> 损伤 -> 寿命
```

未来如需“弹道气动从多物理场预测获得”，不要让 `TrajectoryNode` 和 `FieldPredictionNode` 互相订阅反调。应新增受控编排模型：

```text
CoupledTrajectoryPlanner
  input:
    current_state
    vehicle_profile
    environment_profile
    aero_provider_ref
    coupling_policy

  internal:
    trajectory_model
    aero_provider
    convergence_policy

  output:
    TrajectoryPredictionFrame
    CouplingDiagnosticFrame
```

弹道实时积分不消费完整多场网格，而消费轻量气动提供者接口：

```text
AeroQuery
  time
  position
  velocity
  attitude
  mach
  aoa
  sideslip
  control_surface
  environment

AeroResponse
  force_body
  moment_body
  aero_coefficients
  validity
  confidence
```

耦合策略作为配置：

```text
coupling_policy:
  mode = open_loop | one_shot_correction | fixed_point | offline_plan
  max_iterations
  convergence_tolerance
  timeout_ms
  fallback_to_open_loop
```

第一年建议只做 `open_loop` 和离线 `offline_plan` 小样例，不做在线 fixed-point 闭环。

## 7. 模块拆分和仓库归属

### 7.1 仓库职责

| 仓库 | 新增职责 | 不应承担 |
| --- | --- | --- |
| `flightenv-contracts` | 资源、workflow、状态、轨迹、多场、损伤、寿命、诊断 DTO 和轻量接口 | 不放 DB、ROS2 node、模型实现、文件解析重逻辑 |
| `flightenv-runtime-private` | `platform.sqlite` schema、资源导入、validator、workflow validator、runtime orchestrator、run evidence writer、health ledger writer | 不把这些实现暴露给 UI/SDK，不替代 trajectory 算法仓 |
| `flightenv-trajectory` | 弹道 core、模型 adapter、trajectory package manifest、trajectory node、node smoke | 不暴露 legacy solver，不依赖 runtime-private 私有源码 |
| `flightenv-node-sdk` | `PlatformCatalogReader`、run catalog reader、health reader、orchestration status ViewModel | 不直接访问 runtime-private DB 内部头，不编译 source-supported 给 UI 用 |
| `flightenv-controller-ui` | Twin Workbench 只读展示、运行状态、回放、健康和报告页面 | 不直接解析 `platform.sqlite`、run package binary、trajectory legacy |
| 根 workspace | 文档、根验证、统一 release、跨仓组合验证 | 不拥有子仓实现源码，不承载子仓专用构建逻辑 |

这些是 runtime-private 私有实现。对外只通过 contracts DTO、runtime public artifact、node-sdk 只读接口和 CLI 暴露。

### 7.4 配置边界灰区收口

当前迁移期存在一个需要主动治理的灰区：部分配置类型已经被 SDK 暴露给 UI 使用，而 runtime-private 又把 `EnvPredictorIO/ProjectConfig.hpp` 这类编排头视为私有实现边界。后续资源配置管理不能继续扩大这个重叠面。

收口规则：

```text
稳定语义字段 -> 上移到 flightenv-contracts
路径解析/默认值归一化/preflight -> 留在 flightenv-runtime-private
SDK/UI 读取和展示 -> 使用 contracts DTO 或 node-sdk ViewModel
```

应上移 contracts 的内容：

```text
SourceSpec
SyncSpec
ModelSpecRef
RuntimeProfileDTO
ArchivePolicyDTO
ResourceRefDTO
WorkflowDefinitionDTO
TimeoutPolicyDTO
FailurePolicyDTO
```

应保持 runtime-private 私有的内容：

```text
本机路径搜索
artifact 下载/解压/校验实现
ProjectConfig 到内部 BayesKitConfig 的解析
节点启动顺序
runtime snapshot 等待和错误恢复
OnlineResultWriter / RunManifestWriter 具体实现
```

短期可以兼容现有 SDK 配置入口，但新增资源配置字段应优先按 contracts-first 设计，避免形成第二套 public/private 混合配置面。

## 8. 边界解耦规则

### 8.1 允许的数据流

```text
flightenv-runtime-private -> flightenv-contracts
flightenv-node-sdk        -> flightenv-contracts + runtime public artifact
flightenv-controller-ui   -> flightenv-node-sdk + contracts + runtime artifact
flightenv-trajectory      -> public trajectory headers/lib + root _deps
runtime orchestrator      -> trajectory public API 或 trajectory node topic
```

### 8.2 禁止的数据流

```text
UI -> runtime-private EnvDataBase/EnvPredictorCore/EnvPredictorIO 私有头
UI -> trajectory/legacy/escape/**
node-sdk public headers -> runtime-private 私有类型
contracts -> 任意业务仓
trajectory public headers -> Armadillo/Eigen/BLAS/LAPACK/legacy solver 类型
platform.sqlite -> 存大场数组、模型权重、完整 frame
```

### 8.3 模型耦合规则

允许：

```text
TrajectoryModel -> TrajectoryPredictionFrame -> AeroThermalModel
CoupledTrajectoryPlanner -> AeroProvider interface -> AeroResponse
DamageModel -> DamageIncrement -> HealthAccumulator
```

禁止：

```text
AeroThermalModel 直接 include TrajectoryModel 内部 solver 类
TrajectoryNode 直接 include 多场模型私有实现
LifeNode 反向控制在线 TrajectoryNode
UI 直接选择某个 private model class
```

### 8.4 落盘边界

`RunManifest`、`ResourceLock`、`model_snapshot_json` 必须记录事实，但不复制模型或大数据：

```text
run_<id>/
  manifest.json
  resource_lock.json
  workflow_snapshot.json
  model_snapshot.json
  config_snapshot.json
  runtime_snapshot.json
  index.sqlite
  frames/
  summary/
  logs/
  diagnostics/
```

## 9. 新模块设计

### 9.1 ResourceResolver

输入：

```text
case config
workflow definition
platform catalog
environment variables
artifact roots
```

输出：

```text
resolved ResourceRef list
model package paths
runtime executable paths
dependency roots
storage roots
diagnostic warnings/errors
```

职责：

- 解析相对路径和 artifact 引用。
- 检查 checksum、schema version、manifest。
- 生成 `ResourceLock`。

### 9.2 WorkflowValidator

职责：

- 校验 node/edge 是否完整。
- 校验 `ModelPort` 和 `ModelRelation` 是否匹配。
- 校验 DTO schema、单位、必需字段。
- 校验频率、超时、缓存策略是否合理。
- 在运行前给出错误或 warning。

第一版只做静态 validator，不做动态 DAG executor。

### 9.3 RuntimeOrchestrator

职责：

- 根据固定 workflow 启动和停止节点。
- 维护 topic map。
- 维护 `LatestFrameCache`。
- 监控节点心跳、频率、延迟和异常。
- 触发 `RunManifestWriter` 和 archive。

第一版只支持固定链路：

```text
Sensor -> Filter -> Trajectory
```

第二版扩展：

```text
Sensor -> Filter -> Trajectory -> Field -> Damage -> Life
```

### 9.4 RunEvidenceWriter

职责：

- 写 `manifest.json`。
- 写 `resource_lock.json`。
- 写 `model_snapshot.json`。
- 写 `workflow_snapshot.json`。
- 把 run 注册到 `RunCatalog`。
- 输出 summary 和诊断报告。

### 9.5 PlatformCatalogReader

归属 `flightenv-node-sdk` public API。

职责：

- 只读打开 platform catalog 或 run catalog。
- 返回对象、模型、绑定、run、健康摘要 ViewModel。
- 不暴露 SQLite connection/session。

建议接口形态：

```cpp
class PlatformCatalogReader {
public:
    bool open(const std::filesystem::path& catalog_path);
    std::vector<TwinObjectView> read_objects();
    std::vector<ModelAssetView> read_models();
    std::vector<ModelBindingView> read_bindings();
    std::vector<RunCatalogView> read_runs();
    std::vector<HealthLedgerView> read_health_summary();
};
```

### 9.6 FlightEnvTrajectoryNode

归属 `flightenv-trajectory`。

职责：

- 订阅 `StateEstimateFrame` 或过渡期状态 topic。
- 缓存最新状态。
- 按自身定时器调用 trajectory core。
- 发布 `TrajectoryPredictionFrame` 和 `DiagnosticFrame`。
- 可选落盘轨迹摘要。

不做：

- 不直接 include runtime-private 私有头。
- 不把 ROS2 消息类型带入 trajectory core public API。
- 不执行多场或寿命模型。

## 12. 主要风险和控制

| 风险 | 表现 | 控制 |
| --- | --- | --- |
| 目标膨胀 | 一次性做平台、UI、DAG、寿命闭环 | 每期只交付一个可验收闭环 |
| 过早抽象 | 表很多但没有运行消费 | Trajectory-first pilot，所有 schema 都服务真实 case |
| 状态出口不稳 | 弹道、多场、寿命各自适配滤波结果 | 优先固化 `StateEstimateFrame` |
| 模型关系失控 | 下游 include 上游私有实现 | `ModelPort` + `ModelRelation` + audit |
| 寿命责任过重 | RUL 无验证却被当正式结论 | 显式 confidence、适用包络、unknown/reason |
| 大数据入库失控 | 平台库膨胀、慢、难备份 | 元数据入库，大数据 artifact 引用 |
| UI 穿透私有实现 | UI 直连 DB 或解析 binary frame | node-sdk 只读 reader + 根验收 |
| 多仓同步混乱 | DTO、runtime、SDK、UI 不一致 | 每期明确 owning 仓和验证脚本 |

## 14. 最终判断

这套设计可行，且符合当前多仓拆分和工程成熟度。

关键判断是：

```text
资源配置管理是平台的“事实账本”。
运行时编排是平台的“运行骨架”。
Contracts 是跨仓边界。
runtime-private 是实现主体。
trajectory 是第一个真实模型资产样板。
node-sdk 和 UI 只做只读消费和展示。
```

短期成败不取决于表有多少、UI 有多大，而取决于能否用一个真实链路证明：

```text
当前状态 -> 未来弹道 -> 多场/寿命摘要 -> run 证据 -> 回放解释
```

只要这条链路稳定，后续多模型替换、寿命账本、耦合规划和实验管理都可以按资源、workflow 和 DTO 逐步扩展。
